24 Apr 2013 on design, css

Patterned gradients with CSS pseudo-elements

Neat technique I picked up from Bootstrap's appropriately-named jumbotron element:

div {  
    position: relative;
    width: 100%;
    height: 300px;
    background-color: #1abdbf;
    background-repeat: repeat-x;
    background-image: -moz-linear-gradient(45deg, #104e73, #1abdbf);
    background-image: -webkit-linear-gradient(45deg, #104e73, #1abdbf);
    background-image: -o-linear-gradient(45deg, #104e73, #1abdbf);
    background-image: linear-gradient(45deg, #104e73, #1abdbf);
}

div:after {  
    content: '';
    display: block;
    position: absolute;
    top: 0;
    right: 0;
    bottom: 0;
    left: 0;
    background: url(http://mispy.me/misc/linkypattern.png);
    opacity: .6;
}

Makes a little div with a gradient, then takes a pseudo-element patterened with a background image and squishes them together. The result is something like this (click for jsfiddle):

Since there's currently no equivalent of a layer blend mode for CSS, you cannot (to my knowledge) separate the hue and lightness effects, which limits utility a bit. It helps if you start with a high-contrast image to offset the reduction in opacity. Sort of exchanges visual versatility for technical: you can apply it to arbitrary dimensions and user content, like Subtle Patterns does with its lovely green hover highlights.

17 Feb 2013 on javascript

Aligning points along an SVG path in Javascript

It's possible to create and query a path element without ever attaching it to the DOM, assuming your browser supports SVG.

path = document.createElementNS("http://www.w3.org/2000/svg", 'path')  
path.setAttribute('d', "M150 0 L75 200 L225 200 Z")  
path.getPointAtLength(path.getTotalLength()/2)  
=> { x: 150, y: 200 }

This means you can use the path specification syntax to engage in all sorts of arbitrary bezier curve shenanigans. For example, I made an extremely silly Valentine's Day hack (source). I suspect you could combine this technique with Vector Magic in pursuit of even sillier ends.

06 Jul 2012 on security, chromium

Chromium over a persistent SSH proxy

I often find I need to remotely access web services on ports that can be neither exposed to the outside world nor reverse-proxied to a convenient single location. Fortunately, complex VPN setups are not required here, as the venerable ssh suffices!

-D [bind_address:]port
    Specifies a local “dynamic” application-level port forwarding.  This works by allocating a
    socket to listen to port on the local side, optionally bound to the specified
    bind_address.  Whenever a connection is made to this port, the connection is forwarded
    over the secure channel, and the application protocol is then used to determine where to
    connect to from the remote machine.  Currently the SOCKS4 and SOCKS5 protocols are sup‐
    ported, and ssh will act as a SOCKS server.  Only root can forward privileged ports.
    Dynamic port forwardings can also be specified in the configuration file.

    IPv6 addresses can be specified by enclosing the address in square brackets.  Only the
    superuser can forward privileged ports.  By default, the local port is bound in accordance
    with the GatewayPorts setting.  However, an explicit bind_address may be used to bind the
    connection to a specific address.  The bind_address of “localhost” indicates that the lis‐
    tening port be bound for local use only, while an empty address or ‘*’ indicates that the
    port should be available from all interfaces.

Applying it like so:

ssh -D 11733 some.cool.host

Will give you a SOCKS proxy on local port 11733 for accessing the network environment of some.cool.host. You can persist this proxy by installing autossh alongside screen and adding the following line to your crontab (depending on your authentication method, you might need something fancier):

@reboot screen -S cool_host_proxy -d -m autossh -D 11733 some.cool.host

Since Chromium's settings are somewhat minimalist, you will want an extension to make proxy use more manageable. Proxy SwitchySharp seems to work best under GNOME.

Keep in mind that this will only secure traffic between you and the some.cool.host server. Other means are necessary to prevent snooping between the server and the outside world or by the server itself!

From a semi-random sample of 5000 English tweets, the regex matches 114. Statistical caveats aside, this means that on average the DHS is reading 1 out of every 50 tweets you post.

There are some necessary assumptions made about use of the data here, notably case insensitivity and acknowledgement of word boundaries. I split terms like "Airplane (and derivatives)" into multiple keywords ("airplane", "aeroplane" and "plane") and separated out acronyms. Given the implication that the DHS uses TweetDeck for monitoring, I believe these assumptions are reasonable.
